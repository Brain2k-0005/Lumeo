# Lumeo Docs Nav v2 — Catalog + Sidebar Restructure + Algolia

**Status:** Approved 2026-04-27
**Trigger:** Sidebar holds 128+ component pages today and `Lumeo.Motion` will add 10–30 more. The Form group alone has 34 entries; Data Display has 28. Users "lose the overview." Cmd+K is hand-coded with 514 entries — it does not scale.

## Goal

Three coordinated changes that share one source of truth (`registry.json`):

1. **New `/components` catalog page** — single-page bird's-eye browse, sticky category rail, static thumbnail per component, filter chips.
2. **Sidebar restructure** — subcategorize the bloated Form (34→5) and Data Display (28→5) groups; nav becomes data-driven from a config file instead of hand-coded markup.
3. **Algolia-backed Cmd+K** — replace the 514-entry hand-coded list with live full-content search; existing Cmd+K Dialog UI is retained.

## Non-goals

- `Lumeo.Motion` satellite package (separate spec — depends on this nav landing first).
- AI / RAG search ("ask Lumeo").
- Redesign of individual component pages.
- `BlocksSidebar` redesign — out of scope, untouched.
- Replacing the Cmd+K Dialog UI itself — only its data source changes.

## Architecture

`registry.json` is the single source of truth. The catalog page, the sidebar, the Algolia indexer, and the existing `lumeo` CLI all read from it.

```
                     ┌──────────────────────────────┐
                     │   src/Lumeo/registry.json    │  (existing — extended)
                     │   + new fields:              │
                     │     summary, category,       │
                     │     subcategory, thumbnail   │
                     └──────────────┬───────────────┘
                                    │
        ┌───────────────────┬───────┴────────┬──────────────────────┐
        ▼                   ▼                ▼                      ▼
  NavMenu.razor      Catalog.razor    Algolia indexer         lumeo CLI
  (subcategorized,   (NEW page,       (NEW CI step,          (existing —
   data-driven from   sticky rail,    pushes records         consumes
   nav-config.json)   thumbnails)     to lumeo_docs)         registry)
                                            │
                                            ▼
                                  algolia-search.js
                                  (NEW, called by Cmd+K
                                   in MainLayout.razor)
```

## Surfaces

| Surface | Type | Purpose |
|---|---|---|
| `/components` (`Pages/Catalog.razor`) | NEW | Single-page catalog. Sticky left rail with category jump links + counts. Main scroll area, one `<section>` per category. Filter chips at top. |
| `Shared/CatalogCard.razor` | NEW | One per component. Thumbnail PNG + name + 1-line `summary` + Lucide icon overlay; links to `/components/{slug}`. |
| `Shared/CatalogRail.razor` | NEW | Sticky left rail. Category anchors + per-category counts. Highlights active section via IntersectionObserver. |
| `wwwroot/preview-cards/{slug}.png` | NEW | 480×270 thumbnails, one per component. Generated by the existing `scripts/prerender/preview-card.mjs`. |
| `scripts/prerender/run-all.mjs` | NEW | Batch driver. Iterates `registry.json`, calls `preview-card.mjs` for each entry, writes to `wwwroot/preview-cards/`. Runs in CI. |
| `scripts/algolia/index-docs.mjs` | NEW | Builds records `{objectID, title, summary, category, subcategory, url, type}` from registry + page metadata. Pushes to Algolia index `lumeo_docs`. Runs in CI after `dotnet publish`. |
| `wwwroot/js/algolia-search.js` | NEW | Thin wrapper around the `algoliasearch` JS SDK. Exposes `search(query, opts)` to Blazor via `IJSRuntime`. Reads app ID + search key from a `<meta>` tag injected at build time. |
| `Layout/MainLayout.razor` Cmd+K | MODIFIED | Delete the `_allSearchItems` 514-entry list. `OnSearch` debounces (200 ms) and calls `algolia-search.js`. Renders hits inside the existing Cmd+K Dialog. |
| `Layout/NavMenu.razor` | MODIFIED | Subcategorize Form and Data Display. Replace inline Razor markup with a render loop over `nav-config.json`. |
| `Layout/nav-config.json` | NEW | Single source of truth for sidebar structure (categories, subcategories, item order, icons). |
| `tools/Lumeo.RegistryGen/Program.cs` | MODIFIED | Adds `summary` (scraped from each component's `<PageHeader Description="…">`), `category`, `subcategory` fields to registry entries. |
| `.github/workflows/deploy-cloudflare.yml` | MODIFIED | Add steps: (a) run `scripts/prerender/run-all.mjs` after build, (b) run `scripts/algolia/index-docs.mjs` after publish. |

## Sidebar subcategorization (concrete starting groupings)

Final assignment confirmed at registry-gen time by walking `src/Lumeo/UI/`. The lists below are illustrative starting groupings — the subgroups themselves are the contract; exact membership is finalized in `nav-config.json` during implementation.

**Form (34 → 5 subgroups)**
- Inputs: Input, Textarea, NumberInput, OtpInput, PasswordStrength, MaskedInput, PinInput, SearchInput
- Selection: Select, Combobox, MultiSelect, RadioGroup, CheckboxGroup, ToggleGroup, TreeSelect
- Buttons & Actions: Button, IconButton, ButtonGroup, ToggleButton
- Form Composition: Form, FormField, Label, FieldDescription, FieldError, Fieldset
- Specialized: Slider, Switch, Rating, ColorPicker, DatePicker, TimePicker, RangePicker, FileUpload

**Data Display (28 → 5 subgroups)**
- Tables: Table, DataGrid
- Cards & Layout: Card, KpiCard, StatsCard, Avatar, Badge, AvatarGroup
- Lists & Trees: List, Tree, Timeline, Steps, Carousel
- Charts: existing collapsible subtree — unchanged
- Status & Indicators: Progress, Skeleton, Spinner, Chip, Tag, Delta

Other groups (Layout Primitives, Typography, Feedback, Overlay, Navigation, Utility, AI, Theme) stay flat — none exceed 14 items.

## Data flow

**Build time (CI, in order):**
1. `dotnet build src/Lumeo/Lumeo.csproj` → succeeds.
2. `dotnet run --project tools/Lumeo.RegistryGen` → emits `src/Lumeo/registry/registry.json` with new `summary`, `category`, `subcategory` fields.
3. `node scripts/prerender/run-all.mjs` → reads registry, renders 128 PNGs (parallelized 8 at a time), writes `docs/Lumeo.Docs/wwwroot/preview-cards/`.
4. `dotnet publish docs/Lumeo.Docs/Lumeo.Docs.csproj` → ships.
5. `node scripts/algolia/index-docs.mjs` → reads registry + page metadata → pushes to Algolia index `lumeo_docs` using admin key from `${{ secrets.ALGOLIA_ADMIN_KEY }}`.
6. Cloudflare Pages deploys the published output.

**Runtime:**
- `/components` reads `registry.json` (already an `IRegistryService` injected as singleton). Groups by `category` → renders one `<section id="{slug}">` per category → grid of `CatalogCard`s. Cards lazy-render via `IntersectionObserver` (existing infrastructure used by `BlurFade`).
- `CatalogRail` listens to `IntersectionObserver` on category sections; updates active highlight as user scrolls.
- Filter chips: clicking "Form" hides all sections except `id="form"`. URL syncs to `?filter=form`.
- Cmd+K: typing → debounce 200 ms → `algolia-search.js.search(query)` → Algolia returns up to 8 hits per request → render in the existing Cmd+K Dialog with category badge + type badge (`component | pattern | block | guide`).

## Edge cases

| Case | Behavior |
|---|---|
| Mobile catalog | `CatalogRail` collapses to a horizontal scrolling chip strip pinned under the header. Sections still scroll vertically. |
| Algolia outage / quota exceeded | Cmd+K falls back to a slim local index (component names + categories only) generated alongside `registry.json` and shipped as `_content/Lumeo/registry-search.json`. Better than dead. |
| Missing thumbnail | `CatalogCard` falls back to icon-only treatment using the component's Lucide icon at large size. No broken images, no layout shift. |
| Deep links | `/components#form` scrolls to the Form section and highlights it in the rail. Section IDs use category slugs (`form`, `data-display`, `motion`, etc.). |
| New component added | Author runs `lumeo new MyComponent`, fills `<PageHeader Description="…">`. RegistryGen → prerender → Algolia indexer pick it up automatically — zero manual sidebar edits. |
| Algolia search keys | Public search key + app ID injected at build via `${{ secrets.ALGOLIA_APP_ID }}` and `${{ secrets.ALGOLIA_SEARCH_KEY }}` into a `<meta>` tag in `index.html`. Admin key never reaches the browser. |
| Subcategory disagreement | The starting groupings above are confirmed at registry-gen time by inspecting current page paths. Anything ambiguous defaults to the parent category at the top of its subgroup. Future moves are one JSON edit. |

## Testing

- **Visual:** Snapshot of `/components` at desktop (1440 px) and mobile (375 px). Snapshot of one card variant per category.
- **Functional:** Cmd+K returns at least one result for each known component name within 500 ms (Algolia p99 + 200 ms debounce). Deep link `/components#form` scrolls to the Form section. Filter chip "Form" hides all other sections.
- **CI determinism:** `prerender/run-all.mjs` is deterministic — same `registry.json` input → byte-identical PNG output. PR diffs catch unintended thumbnail regressions.
- **Fallback:** Setting `window.__LUMEO_DISABLE_ALGOLIA = true` in DevTools forces Cmd+K onto the local index; verify it still returns sensible hits for component names.

## Rollout

Single PR, lands behind no flag. Risk is low: NavMenu restructure is reversible, Catalog is purely additive, Cmd+K data swap is one file.

Order within the PR:
1. RegistryGen changes + regenerated `registry.json`.
2. `nav-config.json` + NavMenu refactor.
3. CatalogCard + CatalogRail + Catalog.razor.
4. Prerender batch script + first run of all 128 thumbnails.
5. Algolia indexer + CI step.
6. MainLayout Cmd+K data-source swap + local-index fallback.
7. Workflow YAML changes.
